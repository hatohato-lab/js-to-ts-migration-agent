# js-to-ts-migration-agent 設計メモ

この文書は「**なぜこの設計にしたか**」の記録です。使い方は README を、専門用語を使わない概要は 説明書.md を参照。

## 実証する設計パターン
**差分テスト（differential testing）× EDD**。
移行後コードの正しさを、手書きの期待値でなく「**元のプログラム自身を実行した出力**」で判定する。
LLM 審査や人手に頼らず、機械的・再現可能な合否（外部オラクル）で移行エージェントを評価する。

## なぜ「差分テスト」を選んだか（他方式との比較）★この repo の肝
- この移行は「元の JavaScript が **実行できる**」。だから **元を走らせて、その出力を正解にする**のが最も筋が通る（手書き golden のズレが原理的に消える）。
- 静的 golden（答えを手で保存して比較）は、**元を実行できない**ときの手段。JS は実行できるので、ここでは弱い選択になる。
  → 静的 golden は、Python 2 を実行できない **py2to3-agent** が担当する（役割分担）。
- メタモルフィック（正解を持たず性質で見る）は、正解（元の出力）を作れる本問題ではわざわざ弱める選択。
- LLM 審査は、客観的な正誤がある本問題には不要・過剰。

→ **「元を実行できるか」でオラクルを選ぶ**、という判断の実例。

## なぜ AI-slop にしないか
- 合否が**外部オラクル**（元の実行出力）で決まる。エージェントの自己申告ではない。
- 乱数なし・LLM 審査なしで**完全に決定的**。
- オラクル自身を `--selftest` で検証（正しい移行→PASS／壊れた移行→FAIL）。さらに **控え golden.txt が元の実行出力と一致するか**も点検し、基準が古びていないことを保証する。

## オラクルの 2 段の関門
(a) `tsc --strict` コンパイル → (b) `node` 実行出力が「元の input.js の出力」と一致。

## golden.txt の位置づけ
評価基準は実行で毎回作るので golden.txt は必須ではない。だが「元の出力」を**人が読める形で repo に残す控え**として保持し、selftest ③ で元の実行出力とズレていないか検証する（基準の見える化＋ドリフト検出）。

## corpus
クリーンルームの純粋関数のみ（客先案件の要素ゼロ）:
- 01_fib … 再帰・数値
- 02_fizzbuzz … 制御フロー・文字列配列
- 03_wordcount … オブジェクト/Record・ソート・分割代入

## 一般化
オラクルの「実行器」を差し替えれば他の移行へ広げられる。
ただし **元を実行できる言語**でこそ差分テストが活きる。元を実行できない移行（例: Python 2 → 3）は静的 golden を使う（py2to3-agent）。

## 既知の限界 / 次の拡張
- 出力一致は corpus に用意した入力での「観測可能な振る舞い」しか守らない（全入力の等価保証ではない）。
- corpus が小さい（3 件）。例外・浮動小数・Unicode を足すと弁別力が上がる。
- 入力をランダム生成して差分を取れば、より網羅的な差分テストにできる（拡張余地）。
